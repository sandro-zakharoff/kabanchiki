## 2. Архитектурные решения этой итерации

### 2.1 Хранилище (7.3): Google Drive — осуществимо, интегрируем

Оценка:
- **Авторизация** — OAuth 2.0 личного аккаунта владельца, desktop-flow (loopback + PKCE),
  scope **`drive.file`** (доступ только к файлам, созданным приложением — non-sensitive,
  verification Google не нужна, refresh-токен не протухает у приложения в статусе Production).
- **Раздача клиентам** — дети и Telegram-родители не имеют Google-сессии, поэтому:
  файлы при загрузке получают link-sharing (reader, anyone-with-link, id неугадываемый),
  отображение — через CDN-эндпоинт `drive.google.com/thumbnail?id=…&sz=w480|w1920`
  (быстрый, кэшируемый, без токенов). Приватность обеспечивается вычищенным EXIF/GPS
  и неугадываемыми id — эквивалент нынешних signed URL.
- **Загрузка** — desktop пишет в Drive напрямую своими токенами; Android (дети) и Mini App —
  через новую Edge Function `drive` (проверяет JWT, использует refresh-токен владельца из
  `app_secrets`, тот же паттерн, что telegram_bot_token).
- **Лимиты** — Drive API ~12k запросов/мин на проект, 15 ГБ хранилища: для семьи с фото
  по 200–500 КБ запас огромный. Блокеров нет.

Слой абстракции: у каждого вложения/аватарки поле `storage` (`supabase` | `drive`) +
`path` (путь в бакете или Drive fileId). Чтение всегда работает для обоих бэкендов;
куда писать — решает `app_config.storage_backend`. Drive недоступен → понятная ошибка
и автоматический фолбэк записи в Supabase. Структура папок на Диске:
`Kabanchiki/{tasks,proofs,avatars}`.

### 2.2 Вложения (7.2): таблица `attachments`

`attachments(id, task_id FK cascade, role 'task'|'proof', storage, path, thumb_path,
mime, size_bytes, created_by, created_at)`. Миграция переносит существующие
`tasks.photo_path`/`proof_photo_path` в строки (`storage='supabase'`) без потерь;
старые колонки остаются для совместимости со старыми APK. Дети добавляют пруфы через
SECURITY DEFINER RPC (гард blocked), родители — по RLS. Лимиты: **до 10 фото**,
исходник ≤ 10 МБ — показываются в UI.

### 2.3 Оптимизатор изображений — на клиенте (до загрузки)

Экономит трафик слабых устройств и CPU сервера; сервер принимает готовое.
- ресайз до **1920 px** по длинной стороне;
- **WebP q≈82** там, где кодек есть (desktop Pillow, Android API 30+, Chrome-семейство),
  иначе **JPEG q85** (iOS Safari, Android 8–10);
- миниатюра **480 px q75** отдельным файлом для списков/сеток;
- пере-энкодинг полностью удаляет EXIF (включая GPS), ориентация применяется до ресайза;
- целевой вес: 150–450 КБ полное фото, 20–60 КБ миниатюра.

### 2.4 Аватарки (7.1)

- `profiles.avatar_path/avatar_storage/avatar_updated_at`; файл 512 px WebP/JPEG (~30–80 КБ),
  грузится через тот же слой хранилища в `avatars`.
- **Кто меняет:** администраторы (desktop и Mini App — при создании/редактировании
  исполнителя) **и сам исполнитель** со своего телефона (профиль — это его личность;
  детям это радость, риска нет). Решение зафиксировано.
- Заглушка — инициалы на фирменном фоне (уже есть, полируется).
- Отображение: все места с Avatar/KAvatar/avatar() на трёх платформах.
