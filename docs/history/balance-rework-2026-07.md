Рабочий план, реестр текущего состояния и приёмки. Снапшот до работ: ветка
`backup/pre-balance-rework` (запушена в origin).

> **Промт-источник:** `claude-code-prompt-design-overhaul.md` (переход от
> накоплений внутри задач/работ к личному балансу исполнителя на базе

## 0. Критично: соответствие терминов (промт ↔ код)

Слова заказчика в промте **не совпадают 1-в-1** с именами в коде. Разрешено
однозначно **по поведению**, а не по названию:

| Промт (заказчик) | Поведение | Код / UI |
|---|---|---|
| **«задачи (повременные)»** | почасовые, начисление идёт непрерывно «по мере», есть **вывод из задачи** и поле **«минимальная сумма вывода»** | таблица **`jobs`** / вкладка **«Роботи»** (Works). Только у них есть `min_withdrawal` и `withdrawals` |
| **«задания»** | начисляются **в момент принятия** админом, есть **статус оплаты** | таблица **`tasks`** / вкладка **«Завдання»** (Tasks). Только у них есть `payment_status` |

Признаки, снявшие неоднозначность: `min_withdrawal` и `withdrawals` существуют
**только** у `jobs`; `payment_status` — **только** у `tasks`. Раздел 8 промта
(«одна запущенная задача, двум исполнителям идёт время и счёт») описывает
`job` с двумя `job_members` — это подтверждает: «задача» заказчика = `job` кода.

**Далее в этом документе используются имена кода: `jobs` (Роботи) и `tasks`
(Завдання).**

## 1. Текущая денежная модель (аудит, реестр «как есть»)

### 1.1 Бэкенд (Supabase)

**Таблицы и деньги:**
- `tasks`: `reward_type`(fixed|hourly), `reward_amount`, `earned_amount`,
  `payment_status`(unpaid|awaiting|paid), `paid_at`. Жизненный цикл:
  `new → (timer: in_progress/paused) → submitted → task_review(approve→done /
  reject→declined / rework→new)`. Оплата — отдельная ось, ставится вручную.
- `jobs`: `hourly_rate`, **`min_withdrawal`**. Статусы idle|running|archived.
- `job_members`(job_id, child_id): `earnings_reset_at`, `total_withdrawn`,
  `withdrawn_since_reset`, `joined_at`.
- `job_sessions`: интервалы работы (общий таймер на всех участников).
- `withdrawals`(per job, per child): `amount`, `status`(pending|approved|
  declined), `payment_status`(unpaid|awaiting|paid), `period_from/to`, `paid_at`.
- `bonuses`(child_id, amount, note): ручные начисления.
- `events`: аудит-журнал (AFTER-триггеры на tasks/jobs/job_members/withdrawals/
  bonuses/profiles). Читают только родители.

**Функции (деньги):**
- `job_earned_seconds(job, child)` — секунды с `earnings_reset_at`.
- view `job_member_stats` — `balance = round(earned_seconds/3600*rate,2) −
  withdrawn_since_reset` (баланс живёт **внутри работы**).
- `request_withdrawal(job, amount)` — вывод из конкретной работы, порог
  `job.min_withdrawal`.
- `admin_decide_withdrawal(id, approve)` — full → сброс времени
  (`earnings_reset_at=period_to`, counter=0); partial → `withdrawn_since_reset += amount`.
- `task_complete` → `submitted` (+считает earned).
- `task_review(approve|reject|rework)` — **без начисления денег** (деньги
  «висят» на `earned_amount`, оплата отдельно).
- `task_set_payment`, `withdrawal_set_payment` — ручная простановка оплаты.

**Пуши/уведомления (outbox):** `withdrawal_decided`, `task_reviewed`,
`task_paid`, `withdrawal_paid`, `bonus_granted`, `job_started/stopped`, …

### 1.2 Клиенты (где всплывают деньги)

- **Windows:** `JobDialog.qml`(поле min_withdrawal), `JobsPage.qml`(балансы
  работ), `PaymentBadge.qml`/`PaymentSegment.qml`/`TaskDetailDialog.qml`/

- **Android:** `JobsScreen.kt`(WithdrawSheet, баланс, вывод),
  `JobsRepository.kt`, `Models.kt`(WithdrawalDto, paymentStatus),
  `TaskDetailScreen.kt`(чип оплаты), `BonusesRepository.kt`, `Format.kt`.
- **Telegram Mini App:** `js/app.js`, `js/api.js`(выводы, оплата), `js/format.js`.

### 1.3 Проблемы текущей модели (почему переделываем)
- Деньги «размазаны» по задачам и работам — нет единого личного баланса.
- Вывод привязан к работе; порог задаётся у каждой работы отдельно.
- Оплата задания — ручной статус без движения денег (нет «кошелька»).
- Нет журнала операций как источника истины; баланс — производная от таймеров
  и счётчиков, разбросанных по таблицам; сверить нельзя.

## 2. Целевая модель (ядро)

### 2.1 Личный баланс = append-only журнал операций (ledger)

Новая таблица `ledger_entries` — **единственный** источник изменения баланса.
Никаких изменений баланса мимо неё.

```
ledger_entries(
  id            bigserial pk,
  child_id      uuid,                       -- исполнитель
  amount        numeric(12,2),              -- знаковая: + приход, − расход
  kind          ledger_kind,               -- task | job | bonus | adjustment | withdrawal | reversal
  source_type   text,                       -- 'task'|'job'|'withdrawal'|'manual'
  source_id     uuid,                        -- ссылка на источник
  note          text,
  created_at    timestamptz,                -- реальная дата операции (для миграции — оригинальная)
  actor_kind    text, actor_id uuid, actor_name text,  -- кто инициировал
  dedupe_key    text unique                 -- идемпотентность (см. 2.4)
)
```

**Баланс** исполнителя:
```
balance(child) = Σ ledger_entries.amount(child)  +  live_uncredited_job_accrual(child)
```
- Σ ledger — «закристаллизованная» часть (то, что можно вывести).
- `live_uncredited_job_accrual` — заработок по идущим работам со времени
  последней «сверки» (settlement). Тикает вживую в UI, в журнал попадает
  дискретно (см. 2.2). Для расчёта доступного к выводу — работа сначала
  «сверяется» (settle), т.е. хвост кристаллизуется в ledger.

**Хранение сумм:** `numeric(12,2)` (точная десятичная арифметика Postgres, не
float) — как во всей текущей схеме. Вся денежная арифметика — только на сервере
в SQL `numeric`; клиенты суммы **только форматируют**, не считают.

### 2.2 Работы (jobs / Роботи): авто-начисление на баланс

- Убрать `min_withdrawal`, `withdrawals`-per-job, `total_withdrawn`,
  `withdrawn_since_reset`, `earnings_reset_at`-как-механизм-вывода.
- `job_members` получает `credited_amount numeric(12,2) default 0` —
  **накопительно** сколько уже зачислено в ledger по этой работе этому
  исполнителю.
- `job_member_earned_total(job, child)` = `round(секунды_с_joined_at / 3600 *
  rate, 2)` — точный накопленный заработок с момента входа в работу.
- **Settlement** `settle_job_member(job, child)`: `delta = earned_total −
  credited_amount`; если `delta ≠ 0` → одна запись ledger(kind='job', delta),
  `credited_amount = earned_total`. **Округление не дрейфует**: всегда
  кристаллизуем разницу до точного накопленного значения.
- Settlement вызывается: при **остановке** сессии работы; при удалении
  участника/архивации; при **запросе вывода** (settle-first); плюс
  страховочный pg_cron (например, раз в час) — чтобы ledger был свежим и
  «заработок отражался как поступления».
- Live-хвост (`earned_total_now − credited_amount`) даёт живой тикающий баланс
  без записи в журнал каждую секунду.

### 2.3 Задания (tasks / Завдання): начисление при принятии

- Полностью убрать `payment_status`, `paid_at` (БД, API, UI, все платформы) и
  RPC `task_set_payment`, `withdrawal_set_payment`.
- `task_review(approve)` → одной атомарной операцией создаёт **ровно одну**
  запись ledger(kind='task', amount=earned_amount, dedupe_key='task:'||id).
  Идемпотентно: повтор/сбой не дублируют (unique dedupe_key).
- `reject`/`rework` — денег не двигают. Если approve уже был, а потом задачу
  как-то возвращают — компенсирующая reversal-запись (в норме из `done` назад
  нельзя).

### 2.4 Идемпотентность (никаких двойных зачислений)
- `dedupe_key` UNIQUE: `task:<id>` (принятие), `withdrawal:<id>` (расход),
  `bonus:<id>` (миграция бонусов). Job-accrual — repeatable (dedupe_key NULL),
  но защищён накопительным `credited_amount` (повтор даёт delta=0).
- Все переходы — в одной транзакции с `SELECT … FOR UPDATE` на источнике.

### 2.5 Прочие поступления (бонусы, штрафы, корректировки)
- Ручные корректировки админа — записи ledger(kind='adjustment' | 'bonus'),
  сумма ±, **обязательный комментарий**. Существующие `bonuses` мигрируются в
  ledger (kind='bonus', оригинальные даты). Таблица `bonuses` выводится из роли
  источника баланса (см. миграцию). «Дать бонус» на всех клиентах → «создать
  корректировку (+)».

## 3. Выводы средств — новый флоу

### 3.1 Таблица (пересобираем `withdrawals` под личный баланс)
Убрать `job_id`, `period_*`, `payment_status`. Новые поля:
```
withdrawals(
  id, child_id, amount,
  status   payout_status,    -- requested → approved → paid → confirmed ; + rejected
  method   payout_method,    -- card | cash  (фиксируется при выплате)
  receipt_attachment_id,     -- квитанция (для card), через attachments/файловое хранилище
  reject_reason text,
  comment  text,             -- комментарий к операции («сдача 3 грн за мной»)
  requested_at, approved_at, paid_at, confirmed_at,
  decided_by uuid, ...
)
```

### 3.2 Жизненный цикл и резерв
- **requested**: исполнитель запрашивает сумму из общего баланса. Проверки:
  глобальный минимум (раздел 5) и доступный баланс. Сумма **резервируется**
  сразу debit-записью ledger(kind='withdrawal', −amount, dedupe='withdrawal:'||
  id) — так нельзя запросить больше доступного даже несколькими запросами.
- **approved**: админ одобрил (движения денег нет — уже зарезервировано).
- **paid**: админ отметил выплату. `method` фиксируется здесь.
  - `card`: прикрепляется квитанция (файл/фото) через файловое хранилище
    проекта; открывается из карточки операции. Настройка «квитанция
    обязательна» (раздел 5).
  - `cash`: **двустороннее подтверждение** — админ ставит «выдано наличными»,
    исполнителю уходит запрос подтвердить получение.
- **confirmed**: исполнитель подтвердил получение (терминал для cash; для card
  — `paid`+квитанция терминал, `confirmed` ставится автоматически).
- **rejected**: админ отклонил **с причиной** → компенсирующая credit-запись
  ledger (возврат резерва на баланс). Отмена запроса исполнителем до одобрения
  — так же возврат.
- Долгое неподтверждение налички: операция остаётся `paid`, админ видит
  «очікує підтвердження N днів»; напоминание пушем; ручное закрытие админом с
  пометкой. (Автозакрытие не делаем — деньги требуют явного подтверждения.)

### 3.3 Реестр выводов (Windows) + обработка в Telegram
- Windows: отдельный раздел — список всех выводов, фильтры (исполнитель,
  период, статус, способ), карточка (кто/сколько/когда/способ/квитанция/
  комментарии/история статусов).
- Telegram: уведомления бота на каждый шаг + обработка выводов в Mini App
  (админ), просмотр квитанций.

## 7. Миграция данных (критично, отдельная процедура)
Правила переноса (дефолты заказчика; уточняются по коду):
- Накопленные и не выведенные суммы по **работам** каждого исполнителя →
  на личный баланс записями ledger с оригинальным контекстом и датами
  (`credited_amount` выставляется = перенесённой сумме, чтобы работа не
  начислила повторно).
- Запущенная работа **не прерывается**: время идёт дальше, последующие
  начисления — уже на баланс (settlement по новой схеме).
- **Задания:** принятые+оплаченные (old `paid`) → считаются рассчитанными,
  ledger не трогаем (деньги уже выданы). Принятые (`done`)+не оплаченные →
  начислить на баланс (ledger task-credit). Не принятые — начислятся при
  принятии.
- Незакрытые старые выводы (`pending`/`approved`/`awaiting`) → перенести в
  новый реестр с соответствующим статусом и корректным резервом/движением.
- Вся история видна в новой ленте с оригинальными датами.

Процедура: бэкап БД → прогон на копии → **dry-run отчёт по каждому исполнителю
(сколько и из чего)** → показать заказчику, дождаться подтверждения →
применение в одной транзакции → контрольная сверка (баланс = Σ операций) →
готовый скрипт отката.

### Реализация (готова, gated)
- `20260718100050_balance_migrate_snapshot` — снимок старого состояния ДО того,
  как последующие миграции удалят нужные колонки (payment_status,
  withdrawn_since_reset, старая форма withdrawals). Только чтение, балансы не
  меняет. На пустой БД снимает пусто.
- `20260718100500_balance_migrate_run` — функции (service_role only, НЕ
  запускаются автоматически):
  - `balance_migration_dryrun()` → разбивка по каждому исполнителю
    (работы + задания-неоплаченные + бонусы − резерв выводов = итог);
  - `run_balance_migration()` → постит записи ledger (идемпотентно, с гардом
    ALREADY_RUN), выставляет `credited_amount`/`settled_seconds` работ;
  - `rollback_balance_migration()` → удаляет ровно свои записи, сбрасывает
    базы работ (по таблице `balance_migration_posted`);
  - `balance_migration_verify()` → баланс каждого = Σ его операций.
- Правила переноса: накопленный не-выведенный баланс работ → баланс; принятые
  неоплаченные задания → баланс (оригинальные даты); бонусы → баланс; ожидающие
  (`pending`) выводы → резерв-дебет; одобренные/оплаченные старые выводы уже
  вычтены из старого баланса — повторно не трогаем.

## 8. Приёмка (самопроверка)

## 9. Журнал решений
- **Терминология** разрешена по поведению (раздел 0), имена кода — канон.
- **numeric(12,2)**, не integer-копейки: вся схема уже на numeric (точный
  decimal); переезд на копейки — большой риск-рефактор без выгоды. Арифметика
  только на сервере.
- **Баланс — view + live-хвост**, без преждевременного кэша (семейный масштаб).
  Функция `reconcile` для сверки. Кэш-колонка — как опция, если понадобится.
- **Job-accrual в СЕКУНДАХ (`settled_seconds`), не в накопительных деньгах.**
  Причина: если хранить накопленную сумму, смена ставки работы пересчитала бы
  всё прошлое время по новой ставке (ретро-переоценка). Секунды прайсятся
  порциями по ставке своего периода: settlement кредитует
  `round((elapsed − settled_seconds)/3600 · rate, 2)` и сдвигает
  `settled_seconds`. Дельты никогда не отрицательные; повтор идемпотентен.
  Кристаллизация — на границах: стоп/архив работы, удаление участника, запрос
  вывода (settle-first), смена ставки (триггер), суточный roll-up (cron). Живой
  «хвост» держит отображаемый баланс верным между кристаллизациями.
- **Вывод: резерв debit-записью при запросе** — ledger как единственный
  источник доступного баланса; отклонение → компенсирующий credit.
- **Бонусы → ledger-adjustment**, таблица bonuses уходит из роли источника
  баланса (один источник истины).
- **Разработка на локальном Supabase**, прод — только на гейте миграции.
